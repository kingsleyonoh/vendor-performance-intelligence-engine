# Foundation: Session Auth (UI) vs API Key Auth — Two Separate Surfaces

## What it establishes

VPI has **two distinct authentication surfaces**. They never cross-pollinate.

| Surface | Auth mechanism | Identity set | Controllers |
|---------|---------------|--------------|-------------|
| **UI (`/`, `/dashboard`, `/vendors`, `/session`, `/passwords`)** | Rails 8 built-in session auth (email + password, signed session cookie) | `Current.session` (and `Current.user` via delegation) | Inherit from `ApplicationController`, include `Authentication` concern |
| **API (`/api/*`)** | `X-API-Key` header -> prefix lookup -> SHA-256 compare (`lib/auth/api_key_authenticator.rb`, Phase 1) | `Current.tenant` | Inherit from `Api::BaseController` |

## Files (UI auth)

- `app/models/user.rb` — `has_many :sessions`, `has_secure_password`, normalizes `email_address`
- `app/models/session.rb` — `belongs_to :user`, stores `user_agent` + `ip_address`
- `app/models/current.rb` — `Current.session` + `delegate :user, to: :session`; also carries `tenant` + `request_id`
- `app/controllers/sessions_controller.rb` — `GET /session/new`, `POST /session`, `DELETE /session`
- `app/controllers/passwords_controller.rb` — password reset flow (`/passwords/new`, `/passwords/:token/edit`)
- `app/controllers/concerns/authentication.rb` — `before_action :require_authentication`, `resume_session`, `start_new_session_for(user)`
- `app/mailers/passwords_mailer.rb` + views — reset email
- `test/fixtures/users.yml` — `admin` + `operator` fixtures, BCrypt cost = MIN_COST for speed
- `db/migrate/*_create_users.rb` + `*_create_sessions.rb`

## Files (API auth — built in Phase 1)

- `lib/auth/api_key_authenticator.rb` — Rack middleware
- `app/controllers/api/base_controller.rb` — `require_tenant!` before_action

## Contract

### UI (session) rules

1. Generated by `bin/rails generate authentication` — Rails 8 built-in. **Do NOT add Devise, Rodauth, Clearance, or OmniAuth.** PRD §12 forbids them in v1.
2. UI controllers inherit the `Authentication` concern via `ApplicationController`. Unauthenticated requests redirect to `/session/new`.
3. Public UI routes use `allow_unauthenticated_access only: [...]`.
4. `terminate_session` destroys the DB row AND deletes the cookie. Session rotation on password change is Phase 3.
5. Mailer delivery routes through **Notification Hub** in production (PRD §12: no direct Resend/Twilio). The default ActionMailer adapter is fine for dev.

### API (key) rules

1. Every `/api/*` controller inherits from `Api::BaseController`, which runs `require_tenant!` — fails closed if `Current.tenant` is unset.
2. The `ApiKeyAuthenticator` middleware (Phase 1) is the only component that reads `X-API-Key`. Controllers never touch the header directly.
3. Public `/api` routes (`/api/tenants/register`, `/api/health/*`, `/api/signals/from-hub`) are on an explicit allowlist in the middleware.

### Cross-surface invariants

1. **Never accept a session cookie on `/api/*`.** API clients must send `X-API-Key`. Session auth does not elevate to API access.
2. **Never accept an `X-API-Key` on UI routes.** UI actions are mediated by the authenticated operator + session scope, not by a tenant-wide API key.
3. **`Current.tenant` and `Current.user` can coexist** on a single request only when the UI later integrates tenant-scoped browsing (Phase 1: UI pages scope to the tenant of the logged-in operator via `user.tenant_id`). Until then, UI requests set only `Current.session` / `Current.user`; API requests set only `Current.tenant`.

## When to read this

Before:
- Adding a new UI controller (decide: inherit `ApplicationController`? allow unauthenticated?)
- Adding a new `/api/*` controller (always inherit `Api::BaseController`)
- Wiring the Phase 1 `ApiKeyAuthenticator` middleware
- Changing password policy / session lifetime / cookie flags
- Adding SSO or SAML (forbidden in v1 per PRD §12 — escalate before touching)
