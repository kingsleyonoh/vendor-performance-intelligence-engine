# Foundation: Auth Guard Pattern

## What it establishes

VPI enforces authentication and authorization in **two layers**:

1. **Authentication (who are you?)** — Rack middleware `lib/auth/api_key_authenticator.rb` runs before any controller. It reads `X-API-Key`, resolves the tenant, sets `Current.tenant`. Result: by the time a controller action runs, `Current.tenant` is either set (authenticated) or nil (public allowlist route).

2. **Authorization (what can you do?)** — `ActionPolicy` policies in `app/policies/` run per-action, invoked via `authorize!` in controllers. Base `Api::BaseController` auto-invokes `authorize!` on mutating verbs (POST/PATCH/PUT/DELETE) via `after_action :verify_authorized` equivalent.

The layers do not overlap: the middleware cares only about API keys; policies care only about "does this authenticated tenant/user have permission for this record." Mixing them is the bug class this pattern prevents.

## Files

- `lib/auth/api_key_authenticator.rb` — Rack middleware (future — Phase 1 batch)
- `app/controllers/api/base_controller.rb` — inherits from `ActionController::API`; includes `ActionPolicy::Controller`; defines `render_api_error`; `rescue_from ActionPolicy::Unauthorized` → 403 FORBIDDEN
- `app/policies/application_policy.rb` — base policy (future)
- `app/policies/*_policy.rb` — one per resource

## Contract

### Middleware (authentication)

1. `ApiKeyAuthenticator` intercepts every request. Public allowlist (bypass): `/api/tenants/register`, `/api/health`, `/api/health/*`, `/api/signals/from-hub` (HMAC-verified separately).
2. Flow: read `X-API-Key` header → take first 12 chars as `api_key_prefix` → `Tenant.find_by(api_key_prefix: prefix)` → constant-time `SecureCompare` against `api_key_hash` (SHA-256 of the full key) → set `Current.tenant = tenant`.
3. **Timing-safe comparison mandatory** — `ActiveSupport::SecurityUtils.secure_compare(provided_hash, stored_hash)`. Never `==`.
4. Missing / invalid / unknown key → **401 UNAUTHORIZED** via the foundation error envelope (see `api-error-response-shape.md`).
5. **No cookies, no sessions, no CSRF** for `Api::*` — `ActionController::API` (not `::Base`). Stateless.

### Controller (authorization)

1. Every `Api::*Controller` inherits from `Api::BaseController`.
2. `Api::BaseController` is `ActionController::API` + `include ActionPolicy::Controller`.
3. On any action that requires auth (almost all), call `authorize! record, to: :show?` (or rely on implicit policy lookup). On denial, `ActionPolicy::Unauthorized` is raised → `rescue_from` → **403 FORBIDDEN**.
4. **Cross-tenant access is NOT an authorization failure** — it is handled at the query layer by tenant-scoping (see `tenant-scoping-pattern.md`), which raises `ActiveRecord::RecordNotFound` → **404 NOT_FOUND**. Only use 403 for role/scope within the authenticated tenant.

### UI (separate stack)

The Operator UI (`/dashboard`, `/vendors`, etc.) uses Rails 8 built-in authentication (email + password → cookie session), generated by `bin/rails generate authentication`. That stack lives under `ApplicationController < ActionController::Base` and is orthogonal to the `Api::*` key-based stack. Policies are shared (one `VendorPolicy` serves both).

## When to read this

Before:
- Adding a new API endpoint (must inherit from `Api::BaseController`, must have a policy)
- Modifying the allowlist (any change to what bypasses `ApiKeyAuthenticator` is a security review item)
- Adding a new role / ActionPolicy scope
- Writing a test that asserts auth behavior (401 for missing key, 403 for policy denial, 404 for cross-tenant)
- Onboarding a new ecosystem adapter that needs its own auth path
